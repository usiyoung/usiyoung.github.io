MVC

 |  2022-01-15

들어가기전에

mvc 모델에 대한 정의를 읽던 중 controller가 model, view 사이에서 데이터를 받아 전달하기도 하고 사용자의 url을 받는 역할도 한다는데, 막 와닿지가 않았다. mvc의 view는 프론트를 공부하기 때문에 넘어갔는데 model과 controller 의 역할이 글로 읽어선 잘 구분이 잘 안되었다. 그래서 Codeigniter를 설치해 controller,module,view가 각각 하는 일들을 보여주는 생활코딩의 php codeIgniter편을 정리 해보려한다.

출처: 위키백과

1. Controller

출처: 생활코딩

controller 생성

http://ooo2.org/index.php/topic

위의 url에서 topic이라는 경로를 생성하려면 controllers의 디렉토리 안에 topic.php을 생성해주면 된다.
그 다음에 topic.php파일안에 규칙을 지켜 내용을 채워넣어야 한다.

  1. 경로명과 클래스명은 같아야 하며 대문자로 작성한다.
  2. CI_Controller를 상속한다.

CodeIgniter는 url이 topic/ 뒤로 경로가 없다면 index 메서드를 참조하게 된다. 위의 url을 입력했을 때 나왔으면 하는 페이지가 있다면 index 메서드에 작성해주면 된다.

http://ooo2.org/index.php/topic/get/5

topic 경로 뒤에 get 경로를 사용하려면 아까 작성했던 topic.php파일에 get메서드를 생성하면 된다. 이어 get 경로뒤에 숫자를 입력해 페이지를 변경하고 싶다면, get 메서드의 파라미터에 값을 전달해서 페이지를 바꿔줄 수도 있다. 아래 코드처럼 작성하면 get 메서드의 파라미터 값에 따라 위 url을 입력했을 때 토픽과 아이디가 출력된다.

Controller 개념

  • controller에 의해 url이 작명된다.
  • controller에 의해 웹페이지가 화면에 출력된다.

2. View

출처: 생활코딩

View를 따로 관리해야 하는 이유

위 글을 보면 controller에서도 토픽 페이지라는 텍스트를 출력할 수 있다. 즉, 시각을 담당하는 코드를 controller에 작성할 수 있다. 그럼에도 html,css,javascript와 같은 시각적인 부분을 담당하는 코드는 view 디렉토리에다가 관리해야 한다. 그 이유는 거대한 페이지가 될 수록 코드가 기하급수적으로 늘어나는데 시각을 담당하는 코드,데이터 로직을 담당하는 코드등을 한 곳에 관리할 경우 코드 관리가 어렵기 때문이다. 이러한 이유로 view는 화면에 보여지는 즉, 프론트가 담당하는 코드를 관리하게 된다.

view 생성

view 디렉토리 안에 topic.php를 생성한 후 controller 디렉토리 안에 있는 topic.php으로 돌아가 view를 연결 해준다. 그러면 이번엔 url은 같은 상태에서 controller만을 사용해 화면을 출력 했던 것과 달리 이번엔 controller와 view가 분리된 토픽 페이지가 출력된 화면이 실행된다. 위에서 get 경로를 이용했던 것도 view로 나눠보겠다.

View 개념

  • 사용자에게 시각적인 부분을 보여주는 html,css,javascript 코드를 관리하는 곳이다.

3. Model

출처: 생활코딩

model은 데이터베이스를 담당한다. 데이터를 다루는 로직들을 따로 관리하게 되면서 view와 model을 통해 코드를 뷰와 데이터로 나누어 관리할 수 있기 때문에 코드 관리가 편해진다. 이 글은 mvc 숲의 모양을 이해하지 못해 나무를 살짝 구경하고 있는 중이기 때문에 몇 줄의 코드만 확인하며 controller가 어떻게 model에게 데이터를 받는 지만 확인하겠다.

controller에서 view를 받아오는 것과 동일하게 model도 같은 방식이다. controller 디렉토리의 topic.php에서 index 메서드 안에 작성한 코드이다.

Model 개념

  • 데이터 로직을 모아두는 곳이며, controller에게 요청받은 데이터를 제공해준다.

결론

출처: 생활코딩

  • mvc란 model view controller 의 약자이며, 애플리케이션을 세가지의 역할로 구분한 개발 방법론이다.
  • 사용자가 controller를 조작하면 controller가 그에 맞는 데이터를 model을 통해 가져오고 리턴한 값을 view에 반영해 사용자에게 전달한다. === 사용자가 url을 주소창에 쳤을 때 controller가 model에서 데이터베이스를 받아 view에게 받은 데이터를 기반으로 화면이 사용자에게 보여진다.
  • 사용자로부터 시작해 서버에서 데이터를 받아 뷰로 표현하는 것까지 전반적인 흐름을 코드로 보면서 이해하니 뷰만 공부하다가 눈이 탁 트인 것 같아 흥미롭다. 장황하게는 아니더라도 조금씩 웹 생태계의 견문을 넓히면 뷰,즉 프론트를 설계할 때 좁은 시야로 보는 것에 지나 여러 방면에서 생각하면서 설계할 수 있을 것 같다.
